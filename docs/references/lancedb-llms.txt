# LanceDB quick reference

Used by `src/skill_rag/index.py` as a disposable local cache.

## Commands

- `import lancedb`
- `db = lancedb.connect("./var/index.lance")`
- `tbl = db.create_table("skills", schema=schema)`
- `tbl = db.open_table("skills")`
- `tbl.merge_insert("path").when_matched_update_all().when_not_matched_insert_all().execute(rows)`
- `tbl.search(query_vector).metric("cosine").limit(k).to_list()`
- `db.drop_table("skills")`

## Current Schema Pattern

The table is schema v6:

```python
schema = pa.schema([
    pa.field("path", pa.string()),
    pa.field("name", pa.string()),
    pa.field("description", pa.string()),
    pa.field("content_hash", pa.string()),
    pa.field("text", pa.string()),
    pa.field("agent", pa.string()),
    pa.field("translation_status", pa.string()),
    pa.field("vector", pa.list_(pa.float32(), dim)),
])
```

`path` is the merge key. `text` is the retrieval text used for both embeddings
and BM25. `agent` records the source harness. `translation_status` lets sync
retry unchanged rows whose prior description translation failed or was disabled.

## Constraints

- Treat the LanceDB directory as a cache. Never commit `var/`.
- Schema changes must bump the comment in `src/skill_rag/index.py` and be
  documented in `docs/design-docs/`.
- For small corpora (~50 skills), brute-force search is fine. Add ANN indexes
  only after a measured latency problem.

## Pitfalls

- Schema drift can be handled by dropping and recreating the table because the
  corpus is source of truth.
- Content semantics changes (for example `SkillRecord.embed_text()`) may not
  change schema names. Document `skill-rag reset && skill-rag sync` when needed.
- A long-lived MCP server can hold a model whose vector dimension no longer
  matches the index. `index.search` raises a clear error for this case.
